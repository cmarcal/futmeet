Design Decisions & Notes

ID Strategy

Rooms and games use nanoid (21-char alphanumeric + _-) as primary keys, matching the existing frontend pattern in gameId.ts. Child entities (players, teams, assignments) use Postgres-native UUID via gen_random_uuid().

Normalized vs JSON

Fully normalized tables (not JSON columns) for players and teams. This allows efficient queries for individual player operations (add, remove, reorder, toggle priority) without rewriting entire JSON arrays. Better for Phase 2 real-time broadcasts.

Position Column

All ordered entities use a position INTEGER column to preserve insertion/drag order. On reorder, the API updates affected positions in a transaction. A unique constraint on (parent_id, position) prevents duplicates — use deferred constraints or gap strategy.

Room → Game Flow

POST /rooms/:roomId/start creates a game row (with room_id FK) and copies room_players into game_players. The room persists so the link remains shareable. Games can also be created independently (POST /games with room_id = NULL).

Cascade Deletes

All child FKs use ON DELETE CASCADE so removing a room/game/team automatically cleans up its children. The games.room_id FK uses ON DELETE SET NULL so a game survives if its source room is deleted.

Game Status

Stored as VARCHAR(20) with a CHECK constraint instead of Postgres ENUM to simplify migrations when new statuses are added. Values: 'setup' (default), 'sorting' (during sort), 'complete' (teams assigned).

Team Count Validation

Database-level CHECK (team_count BETWEEN 2 AND 10) matches the frontend Zod schema. The API also validates before reaching the DB, but the constraint acts as a safety net.

Timestamps

All tables use TIMESTAMPTZ (timestamp with time zone). The API returns ISO 8601 strings. Parent tables (rooms, games) include updated_at for cache invalidation. Child tables omit it for simplicity (only created_at).

Indexes

Foreign key columns (room_id, game_id, team_id) are indexed for fast JOINs and lookups. Postgres does NOT auto-index FKs (unlike MySQL), so explicit indexes are essential.

ROOMS CREATE TABLE rooms ( id VARCHAR(21) PRIMARY KEY, created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(), updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW() ); -- 2. ROOM PLAYERS CREATE TABLE room_players ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), room_id VARCHAR(21) NOT NULL REFERENCES rooms(id) ON DELETE CASCADE, name VARCHAR(50) NOT NULL, priority BOOLEAN NOT NULL DEFAULT FALSE, position INTEGER NOT NULL, notes TEXT, created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(), UNIQUE (room_id, position) ); CREATE INDEX idx_room_players_room_id ON room_players (room_id); -- 3. GAMES CREATE TABLE games ( id VARCHAR(21) PRIMARY KEY, room_id VARCHAR(21) UNIQUE REFERENCES rooms(id) ON DELETE SET NULL, team_count INTEGER NOT NULL DEFAULT 2, game_status VARCHAR(20) NOT NULL DEFAULT 'setup', created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(), updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(), CONSTRAINT chk_game_status CHECK (game_status IN ('setup', 'sorting', 'complete')), CONSTRAINT chk_team_count CHECK (team_count BETWEEN 2 AND 10) ); -- 4. GAME PLAYERS CREATE TABLE game_players ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), game_id VARCHAR(21) NOT NULL REFERENCES games(id) ON DELETE CASCADE, name VARCHAR(50) NOT NULL, priority BOOLEAN NOT NULL DEFAULT FALSE, position INTEGER NOT NULL, notes TEXT, created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(), UNIQUE (game_id, position) ); CREATE INDEX idx_game_players_game_id ON game_players (game_id); -- 5. TEAMS CREATE TABLE teams ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), game_id VARCHAR(21) NOT NULL REFERENCES games(id) ON DELETE CASCADE, name VARCHAR(50) NOT NULL, position INTEGER NOT NULL, UNIQUE (game_id, position) ); CREATE INDEX idx_teams_game_id ON teams (game_id); -- 6.
